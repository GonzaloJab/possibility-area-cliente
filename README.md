# Possibility® — Área Cliente

Energy invoice management dashboard for Possibility® clients.

**Stack**
- **Frontend:** Vite + React + TypeScript (apps/web)
- **Backend:** FastAPI + SQLAlchemy 2 (async) + Alembic + JWT (apps/api)
- **Database:** Postgres (Neon in prod, local Postgres for dev via Docker)
- **Storage:** Cloudflare R2 for invoice PDFs
- **Deploy:** Render (Docker), GitHub Actions builds & pushes images to GHCR
- **Portable:** Same Dockerfiles run locally, on Render today, and on Hetzner tomorrow

---

## 1. Local development

You only need **Docker Desktop** and **git** installed locally.

```bash
cp .env.example .env       # adjust if needed; defaults work out of the box
docker compose up --build  # spins up Postgres + FastAPI + React (nginx)
```

- Frontend: <http://localhost:5173>
- API docs: <http://localhost:4000/docs>
- Demo login: `felipe@possibility.com` / `possibility123`

The api container automatically runs `alembic upgrade head` and seeds 3 supplies on first boot.

### Iterative dev (without Docker)
If you'd rather run services natively for faster reloads:

```bash
# 1. Start just Postgres in Docker
docker compose up -d postgres

# 2. API (Python 3.12)
cd apps/api
python -m venv .venv && source .venv/bin/activate     # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp ../../.env.example .env
alembic upgrade head
python seed.py
uvicorn app.main:app --reload --port 4000

# 3. Web (Node 20+) — in another terminal
cd apps/web
npm install
echo "VITE_API_URL=http://localhost:4000" > .env
npm run dev
```

### Database migrations

```bash
cd apps/api
alembic revision --autogenerate -m "add new field"   # after editing app/models.py
alembic upgrade head                                  # apply migrations
alembic downgrade -1                                  # roll back last migration
```

## 2. Production deploy (Render + Neon + R2)

### 2.1 Provision Neon Postgres
1. Sign in to <https://console.neon.tech> with GitHub.
2. Create project: **Name** = `possibility-area-cliente`, **Region** = AWS eu-central-1 (Frankfurt).
3. Copy the **pooled connection string** from Dashboard → Connection Details.
4. Two URLs are needed:
   - `DATABASE_URL` — `postgresql://USER:PASS@ep-xxx.eu-central-1.aws.neon.tech/possibility?sslmode=require`
   - `DATABASE_URL_ASYNC` — same string but with `postgresql+asyncpg://` prefix and `?ssl=require`

### 2.2 Provision Cloudflare R2 bucket
1. Sign in to <https://dash.cloudflare.com> with GitHub.
2. R2 → Create bucket → name `possibility-invoices` → location: Automatic.
3. Manage R2 API Tokens → Create API Token → Object Read & Write to this bucket.
4. Copy: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`.
5. Settings → Public Access → enable a public r2.dev URL (or attach a custom domain). Copy as `R2_PUBLIC_URL`.

### 2.3 Connect GitHub repo to Render
1. Push this repo to GitHub (see §3 below).
2. Render Dashboard → New → Blueprint → connect this repo.
3. Render will read `render.yaml` and create both services (`possibility-api`, `possibility-web`).
4. Fill the secrets it asks for:

| Service | Variable | Value |
|---|---|---|
| api | `DATABASE_URL` | (Neon pooled, with `?sslmode=require`) |
| api | `DATABASE_URL_ASYNC` | (same, with `postgresql+asyncpg://` and `?ssl=require`) |
| api | `CORS_ORIGINS` | `https://possibility-web.onrender.com` |
| web | `VITE_API_URL` | `https://possibility-api.onrender.com` |

`JWT_SECRET` is auto-generated by Render. Add R2 vars later when you wire PDF uploads.

### 2.4 Auto-deploy

`render.yaml` sets `autoDeploy: true`. Every push to `main` rebuilds both services on Render, and the GitHub Actions workflow publishes images to `ghcr.io/<your-repo>/{api,web}:{sha,latest}` in parallel.

## 3. Git + GitHub setup

```bash
git init
git add .
git commit -m "Initial scaffold: FastAPI + React + Docker + CI/CD"
git branch -M main
git remote add origin git@github.com:<you>/possibility-area-cliente.git
git push -u origin main
```

Required repo settings:
- **Settings → Actions → General**: allow GITHUB_TOKEN to write packages (default on public repos).
- **Settings → Branches**: protect `main`, require CI to pass before merge.
- **Settings → Secrets**: only needed if you add a Render deploy webhook (optional — Render auto-deploy already triggers on push).

## 4. Migrating to Hetzner later

When you outgrow Render's free tier:

1. Spin up a Hetzner CX22 (€4/mo) and install Docker + Docker Compose.
2. Copy the existing `docker-compose.yml`, replace the `build:` blocks with `image: ghcr.io/<you>/possibility-{api,web}:latest`.
3. Set up Caddy or Traefik for HTTPS termination + reverse proxy.
4. Switch DNS A records.
5. The Neon DB and R2 bucket stay where they are — no data migration needed.

Or use Coolify / Dokploy on the Hetzner box for a one-click "PaaS-on-your-VPS" experience.

## 6. API reference (auth required except `/auth/login`, `/auth/register`, `/health`)

| Method | Endpoint | Description |
|---|---|---|
| POST | `/auth/register` | Create account |
| POST | `/auth/login` | Returns `{ token, user }` |
| GET | `/auth/me` | Current user |
| GET | `/supplies` | List user's supplies |
| GET | `/supplies/{id}` | Full detail (tariff + invoices + consumption) |
| GET | `/supplies/{id}/invoices` | Invoices only |
| GET | `/supplies/{id}/consumption` | Consumption history |
| GET | `/supplies/{id}/tariff` | Tariff details |

Interactive docs at `/docs` (development only).
